### 查询数据流

```

  │
  ▼

  │
  ├── Step 1: 意图识别 (LLM 分类 + 置信度)
  │     └── 置信度 < 0.7 → LLM 追问确认 → 结束
  │
  ├── Step 2: 问题重写 (LLM 代词消解)
  │
  ├── Step 3: 检索缓存检查 (Redis 余弦相似度)
  │     └── 命中 → 注入上一轮缓存块
  │
  ├── Step 4: 工具选择 (LLM 决定调用哪些工具)
  │     ├── retrieve_knowledge → 执行 → 来源去重 → 发射 SSE
  │     ├── query_documents → 执行 → 保存文档上下文到 Redis
  │     ├── get_current_time → 执行
  │     └── LLM 未选工具 → jieba 实体检查 → 强制检索
  │
  ├── Step 5: 记忆加载 (短期摘要 + 长期记忆)
  │
  ├── Step 6: 上下文组装 (构建 system prompt)
  │
  ├── Step 7: LLM 流式生成 (SSE, 可选深度思考/CoT)
  │
  ├── Step 8: 标题生成 (自动 LLM 生成, 仅首轮)
  │
  └── Step 9: 后处理 (持久化对话、审计日志)
```

### 文档上传数据流

```
上传 → API (MIME 校验 + MD5) → MinIO (原始文件)
  → PG (文档记录, status=pending)
  → 队列 (内存或 RabbitMQ) → Worker
    → 解析 (MinerU/PyMuPDF)
    → 分块 (语义 + 标题跟踪)
    → 增强 (HQG + 摘要 via LLM)
    → 向量化 (bge-m3 → Milvus, 按 dept_id 路由)
    → PG (status=ready, chunk 记录, tsvector 更新)
```

## 2. 多部门数据隔离

### 问题

企业系统要求不同部门之间严格数据隔离。研发部门的用户不能查看 HR 部门的文档。

### 方案：Milvus 字段过滤隔离

开源版使用**字段级隔离**：向量存储在共享 collection 中，通过 `dept_id` 字段过滤。

```
Collection: knowledge_chunks

集合内：
├── Record {dept_id: "d1", ...}   ← 部门 A
├── Record {dept_id: "d2", ...}   ← 部门 B
└── ...

检索 = 字段(dept_id) 过滤
```

部门隔离通过 `dept_id` 字段表达式实现：

```python
search_params = {
    "expr": f"dept_id in ['{dept_id}']",
    "limit": top_k,
}
```

### 管理员

管理员 `dept_id = None`，检索时跳过 dept 过滤：

```python
if current_user.is_super_admin:
    search_params = {"limit": top_k}
else:
    search_params = {
        "expr": f"dept_id in [{user_dept_ids}]",
        "limit": top_k,
    }
```

### 三层隔离

| 层 | 机制 | 范围 |
|-------|-----------|-------|
| **API** | RBAC 中间件 | 用户可执行的操作 |
| **向量** | Milvus 字段过滤 | 用户可搜索的向量数据 |
| **应用** | SQL `WHERE dept_id = ?` | 用户可读取的记录 |
| **文件** | MinIO 路径前缀 (`raw/{dept_id}/`) | 用户可访问的文件 |

## 3. RAG 管线

### 索引管线

文档经过多阶段处理以确保检索质量：

```
上传 (.pdf/.docx/.xlsx/.pptx/.md/.txt/.csv/.msg/.eml)
  │
  ▼
[解析器] — PyMuPDF (PDF，字体大小检测标题)
           python-docx (DOCX，标题层级感知)
           python-pptx (PPTX，幻灯片标题+正文)
           openpyxl (XLSX，工作表感知)
           Native 解析 (TXT/MD/CSV)
           email.parser (MSG/EML)
  │
  ▼
[分块器] — 标题感知分块（无标题时降级）：
            - 按 Markdown 标题分割
            - 无标题 → 按段落 → 按句子递归分块
            - 跨 chunk overlap 保持上下文连续
            - 合并小段落
  │
  ▼
[索引引擎] — 每个分块并行处理 (3s 超时)：
            ├── 摘要：LLM 生成简洁摘要
            └── HQG：LLM 生成假设问题
  │
  ▼
[Milvus] — HuggingFaceEmbedding(bge-m3, 1024d) via LlamaIndex
           插入 dept_id + project_id + visibility 字段
  │
  ▼
[PostgreSQL] — 写入 chunk 内容 + heading_path + page_range
               更新 content_tsv (tsvector 全文索引)
               状态: ready
```

### 检索管线

> 

查询时管线支持可选策略 + 标准多路检索：

```
                          用户查询
                              │
              ┌───────────────┴───────────────┐
              │  可选前置策略:                  │
              │  ├─ HyDE: LLM 生成假设答案 →    │
              │  │   embedding → 向量检索      │
              │  ├─ QueryFusion: 多视角查询     │
              │  │   变体 → 并行检索 → RRF      │
              │  └─ StepDecomp: 子问题分解 →    │
              │     顺序检索 → 聚合             │
              └───────────────┬───────────────┘
                              │
                ┌─────────────┴─────────────┐
                │                           │
    ┌───────────▼───────────┐   ┌───────────▼────────────┐
    │  Route 1: 向量检索     │   │  Route 2: 全文检索       │
    │  (pymilvus)           │   │  (PostgreSQL tsvector) │
    │                       │   │                        │
    │  1. HuggingFaceEmbed  │   │  1. tsquery 匹配        │
    │     (bge-m3, 1024d)   │   │  2. ILIKE 降级          │
    │     via LlamaIndex    │   │     (中文 jieba 分词)    │
    │  2. dept_id + visibi- │   │  3. dept_id 过滤        │
    │     lity filter 表达式 │   │                        │
    │  3. 余弦相似度 Top-K    │   │                        │
    │     (IVF_FLAT)        │   │                        │
    └───────────┬───────────┘   └───────────┬────────────┘
                │                           │
                └─────────────┬─────────────┘
                              │
                     ┌────────▼────────┐
                     │  RRF 融合        │
                     │  (k=60)         │
                     └────────┬────────┘
                              │
                     ┌────────▼────────┐
                     │  BM25 重排       │
                     │  (jieba + BM25) │
                     │  防 RRF 摊平     │
                     └────────┬────────┘
                              │
                     ┌────────▼────────┐
                     │  Cross-Encoder  │
                     │  重排            │
                     │  (bge-reranker) │
                     │  via LlamaIndex │
                     └────────┬────────┘
                              │
                     ┌────────▼────────┐
                     │  严格模式        │
                     │  (score<0.2)    │
                     │  → "未找到"      │
                     └────────┬────────┘
                              │
                     ┌────────▼────────┐
                     │  Top-K 结果      │
                     │  → 上下文组装     │
                     │  → LLM 生成      │
                     └─────────────────┘
```

### 为什么多路？

| 路线 | 优势 | 劣势 |
|-------|----------|----------|
| **向量** | 语义理解，可处理同义改写 | 精确关键词匹配弱 |
| **全文** | 精确关键词匹配，专有名词 | 无语义理解 |

两路互补。RRF 融合提升多路命中文档的排序，来源去重减少噪声，Cross-Encoder 做最终质量把关。

### 意图分类

六种意图：

| 意图 | 动作 |
|--------|--------|
| `knowledge_query` | 检索知识库后生成 |
| `general_chat` | 跳过检索，直接 LLM 回复 |
| `document_task` | 查询文档信息 |
| `harmful` | 安全门控触发，LLM 决定回复 |
| `out_of_scope` | 安全门控触发，LLM 决定回复 |
| `sensitive_topic` | 安全门控触发，LLM 决定回复 |

每个意图带有**置信度评分**。当置信度 < 0.7 时，系统追问确认而非猜测。

### 安全门控系统

门控根据意图标记安全标志：

```python
GUARDRAIL_MAP = {
    "harmful":       ["harmful", "safety"],
    "out_of_scope":  ["out_of_scope"],
    "sensitive_topic": ["sensitive"],
}
```

被标记的意图向 system prompt 注入安全指令。工具结果可覆盖门控——若 LLM 已选择工具，即使意图被标记也视为合法。

### SSE 事件流

响应通过 Server-Sent Events 推送：

```
POST /api/v1/chat/message
Accept: text/event-stream

event: intent
data: {"intent": "knowledge_query", "reasoning": "..."}

event: retrieval
data: {"chunks": [...], "queried_doc_ids": [...]}

event: thinking              (深度思考模式)
data: {"text": "让我逐步分析这个问题..."}

data: {"text": "根据"}

data: {"text": "知识库记录，"}

event: token
data: {"text": "Project Alpha 由张伟负责。"}

event: done
data: {"message_id": "...", "session_id": "...", "usage": {}}
```

## 5. 记忆系统

### 短期记忆（会话上下文）

- **存储**：Redis AOF（崩溃恢复）
- **结构**：
  - `session:{id}:messages` — 最近 N 条消息列表
  - `session:{id}:summary` — 压缩摘要（超过 token 阈值时重新生成）
- **TTL**：30 分钟滑动窗口（每次交互重置）
- **摘要触发**：当总 token 超过上下文窗口的 75%

### 长期记忆（跨会话）

- **存储**：PostgreSQL `user_memories` 表
- **提取**：LLM 评估每条用户消息，提取"值得记住"的事实
- **检索**：通过 bge-m3 对记忆内容做语义搜索
- **写入策略**：异步后台任务，不阻塞主流程

### 对话历史

- PostgreSQL 存储，带 `metadata` JSONB 字段
- 支持逐条消息软删除
- 首条消息由 LLM 自动生成标题

## 6. 安全模型

### RBAC 架构

```
用户 → 角色 → 权限
  │       │        └── resource:action (如 "document:upload")
  │       └── 按 dept_id 限定范围
  └── 绑定到部门
```

### 角色体系

| 角色 | 知识库 | 问答 | 系统设置 | 用户管理 |
|------|--------|------|---------|---------|
| **admin** | 全部部门 | 可用 | 全部 | 全部 |
| **dept_admin** | 本部门 CRUD | 可用 | 部分 | 本部门 |
| **dept_editor** | 本部门 CRUD | 可用 | — | — |
| **dept_viewer** | 本部门只读 | 可用 | — | — |

### 权限实施

```python
# 路由层装饰器
@router.post("/documents")
@require_permission("document:upload")
async def upload_document(...):
    ...

# 中间件层角色检查
@router.delete("/admin/users/{id}")
@require_role(["admin", "dept_admin"])
async def delete_user(...):
    ...
```

### 安全措施

| 措施 | 实现方式 |
|---------|---------------|
| **认证** | JWT Bearer token，定期轮换 |
| **密码** | bcrypt 哈希 |
| **限流** | Redis 滑动窗口，每用户 + 每 IP |
| **文件上传** | MIME 类型校验，50MB 限制，禁止可执行文件 |
| **暴力破解** | N 次失败后账户锁定（可配置） |
| **CORS** | 白名单域名验证 |
| **审计** | 所有操作记录 trace_id |
| **SQL 注入** | 仅 ORM 查询 (SQLAlchemy) |
| **日志** | JSON 结构化日志，不记录敏感数据 |

## 7. API 契约架构

### 设计目标

前端不绑定后端实现，可替换为 Java/Spring Boot（信创场景）。

### 工作原理

```
React 前端
  │
  ├──► http://api-server/api/v1/...  ← OpenAPI 契约
  │
  ├──► Python/FastAPI 后端（当前）
  └──► Java 后端（未来信创）
```

### 契约边界

| 关注点 | 前端 | 后端 |
|---------|----------|---------|
| 路由 | react-router-dom | FastAPI routes |
| 状态 | Zustand stores | — |
| 认证 | JWT 存储 + Bearer 头部 | JWT 签发 + 验证 |
| 数据 | 渲染 API 响应 | SQLAlchemy 模型 |
| 业务逻辑 | UI 状态管理 | 核心领域逻辑 |
| 存储 | — | 全部数据库 |

### API 约定

```
Base URL: /api/v1/
格式: JSON
认证: Bearer JWT
响应: {code, message, data, meta}
错误: {code, message, detail}
分页: {page, page_size, total, items}
```

契约先行模式的优势：
- 前后端可并行开发
- 后端可重写而不改 UI
- API 变更显式版本化

## 8. 关键技术决策

### 为什么选择单体而非微服务？

| 因素 | 单体 | 微服务 |
|--------|----------|---------------|
| 部署复杂度 | 一个服务 | 多个服务 |
| 开发速度 | 快速迭代 | 协调开销 |
| 团队规模 | 一个团队 | 需要多个团队 |
| API 契约迁移 | 容易（进程内） | 需 API 网关 |

**决策**：从结构清晰的单体开始。API 契约层保证前后端解耦。如果未来需要，模块可独立提取为服务。

### 为什么选 Milvus 而非 PGVector？

| 因素 | Milvus | PGVector |
|--------|--------|----------|
| 大规模性能 | 优化百万级以上 | 高并发下降 |
| 功能集 | 丰富索引类型、标量过滤 | 基础向量操作 |
| 部署复杂度 | 独立服务 | 数据库内置 |

**决策**：需要硬隔离和性能的场景使用 Milvus。小规模部署可用 PGVector 替代。

### 为什么选 LangGraph 而非 LangChain Agent？

| 因素 | LangGraph | LangChain Agent |
|--------|-----------|-----------------|
| 状态控制 | 显式状态机 | 隐式工具选择 |
| SSE 流式 | 细粒度控制 | 粗粒度 |
| 调试 | 确定性节点执行 | 非确定性 |
| 意图路由 | 自定义分类 | LLM 驱动工具选择 |

**决策**：LangGraph 的显式状态机提供更好的控制、可预测性和 SSE 事件分发。

### 为什么 RRF + Cross-Encoder 而非学习排序？

| 因素 | RRF + Cross-Encoder | 学习排序 |
|--------|---------------------|-----------------|
| 实现 | 简单，零训练 | 需要训练数据 |
| 质量 | 接近学习排序 | 数据充分时更优 |
| 维护 | 无需模型更新 | 需要定期重训练 |
| 冷启动 | 立即可用 | 需要历史数据 |

**决策**：RRF + Cross-Encoder 零训练开销即可获得出色效果。查询量和数据增长后可替换为学习排序。
